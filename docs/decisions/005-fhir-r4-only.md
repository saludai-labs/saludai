# ADR-005: FHIR R4 Only

**Status:** Accepted

## Context

HL7 FHIR has multiple published versions:
- **DSTU2** (v1.0, 2015): first widely implemented version
- **STU3** (v3.0, 2017): significant improvements, still in use
- **R4** (v4.0, 2019): first "normative" version — backward compatibility guaranteed
- **R5** (v5.0, 2023): latest version, early adoption
- **R6** (in development): draft

SaludAI needs to interact with FHIR servers in the public health ecosystem of Argentina and Latin America. We must decide which FHIR versions to support.

## Decision

Support **exclusively FHIR R4** (v4.0.1). No support for DSTU2, STU3, R5, or R6.

This applies to:
- The FHIR client (`saludai-core`)
- Data models and validations
- Queries generated by the agent
- The HAPI FHIR development server
- Synthetic seed data

## Consequences

### Positive
- **Simplicity:** a single set of resource schemas, search parameters, and behaviors
- **Argentine ecosystem:** openRSD (Argentina's national profiles) uses FHIR R4
- **HAPI FHIR:** the reference server has its best support for R4
- **Python libraries:** `fhir.resources` has complete and stable R4 support
- **Stability:** R4 is normative — resources won't change in breaking ways
- **Reproducible benchmarks:** a single target simplifies agent evaluation

### Negative
- **No R5 support:** some new R5 features (SubscriptionTopic, etc.) are unavailable
- **Legacy excluded:** servers that only speak DSTU2/STU3 are incompatible
- **Future migration:** if LATAM adopts R5 massively, migration work will be required

### Risks
- If an important health system requires R5, we can't connect without additional work
- Mitigation: the `FHIRClient` abstraction can be extended to support multiple versions; R4→R5 has high compatibility in common resources (Patient, Condition, Observation)

## Alternatives Considered

### Option A: Multi-version (R4 + R5)
- Pros: greater compatibility, future-proof
- Cons: double maintenance of schemas, search params, and validations; significant complexity in the agent loop to handle version differences

### Option B: R5 only
- Pros: newest version, better features
- Cons: minimal adoption in LATAM, openRSD has no R5 profiles, HAPI FHIR R5 is less tested

### Option C: DSTU2 + R4 (maximum legacy compatibility)
- Pros: supports legacy servers
- Cons: DSTU2 is fundamentally different in structure; maintaining two data models is infeasible for a small team
