# ADR-008: FHIR Awareness in Locale Packs

**Status:** Accepted

## Context

Locale packs (ADR-007) cover terminology and UI localization (prompts, descriptions), but don't address structural FHIR customizations that each country/implementation defines: profiles, extensions, custom operations, search parameters, identifier systems, and validation rules.

FHIR is extensible by design. Each national implementation (like openRSD/AR.FHIR.CORE in Argentina) defines profiles that add required fields (e.g., DNI), custom extensions (e.g., ethnicity, NOMIVAC scheme), and country-specific identifier systems (RENAPER, REFEPS, REFES).

For the agent to be truly adaptable to different countries, it needs to *know* about these local customizations.

## Decision

Implement FHIR awareness in two levels:

### Level 1 — Awareness (implemented)

Extend `LocalePack` with declarative metadata about the local FHIR implementation. The agent receives this metadata as part of the system prompt (dynamically generated by `build_fhir_awareness_section`). The agent *knows* that profiles, extensions, identifiers, etc. exist and can mention them, but does not validate against them.

New types:
- `FHIRProfileDef` — profiles with required extensions
- `ExtensionDef` — FHIR extensions with URL, type, context
- `IdentifierSystemDef` — identifier systems (DNI, CUIL, etc.)
- `CustomOperationDef` — custom FHIR operations ($summary, etc.)
- `CustomSearchParamDef` — custom search parameters
- `LocaleResourceConfig` — how each resource is used locally

### Level 2 — Execution (planned)

The agent validates responses against profiles, uses custom operations, and parses extensions intelligently. Planned for a future phase.

## Consequences

### Positive
- The agent knows the local FHIR context without hardcoding in the prompt
- Adding a new country is declarative: define metadata, the prompt generates itself
- Backward compatible: all new fields have empty defaults
- Clear differentiator: "adaptable to any national FHIR implementation"

### Negative
- The system prompt grows (~50 extra lines for AR) — more tokens per request
- Metadata can become outdated if the local IG changes

### Risks
- Level 2 requires access to real FHIR servers with profiles/extensions for testing validation

## Alternatives Considered

### Option A: Hardcode in the prompt
- Pros: simple, fast
- Cons: doesn't scale, not reusable, can't generate per-country prompts

### Option B: Load StructureDefinitions from the FHIR server
- Pros: always up to date
- Cons: requires connection, slow, complex to parse, not available offline

### Option C: Declarative metadata in locale pack (chosen)
- Pros: offline, versionable, simple, extensible
- Cons: can become outdated
